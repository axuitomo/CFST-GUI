# CFST-GUI 产品介绍

## 1. 文档定位

本文面向第一次接触 CFST-GUI 的用户，目标是用尽量通俗的方式说明：

- 这个产品是做什么的
- 它解决了什么实际问题
- 适合哪些人使用
- 应该下载哪个发行版
- 如何通过发行版或 Docker 完成安装与部署

如果你只想直接使用成品，而不是参与源码开发，这份文档就是为你准备的。

---

## 2. 产品简介

CFST-GUI 是一个面向 Cloudflare/CDN 场景的 IP 测速与筛选工具。它把原本偏命令行、偏工程化的测速流程，整理成了一个可视化界面，帮助用户更容易地完成以下工作：

- 导入候选 IP 或域名列表
- 执行延迟、丢包、追踪、下载速度等测试
- 按条件筛选出更合适的目标节点
- 导出结果 CSV
- 在需要时读取 Cloudflare DNS 记录，并通过自动推送或工作流把结果同步到 Cloudflare DNS

它的产品形态并不只有一种：

- Windows、macOS：提供桌面发行版，下载后即可直接运行
- Linux：提供 WebUI Docker Compose 部署包，适合服务器、NAS、小主机等长期运行环境
- Android：提供 universal、arm64-v8a 和 armeabi-v7a APK 发行包，并通过应用内更新 manifest 匹配当前设备 ABI

对大多数普通用户来说，可以把它理解成：

> 一个“帮你从一批 Cloudflare/CDN 候选 IP 里挑出更适合当前网络环境那一批”的可视化工具。

---

## 3. 项目背景

在 Cloudflare 或其他 CDN 场景里，很多用户都会遇到一个共同问题：同样的服务域名，在不同地区、不同运营商、不同时间段下，实际访问质量会明显不同。

这背后的原因通常不是“服务坏了”，而是链路质量在波动，例如：

- 某些入口 IP 延迟偏高
- 某些节点存在丢包或回源不稳定
- 某些 IP 的下载速度不理想
- 同一个服务，在白天和晚上表现差异明显
- 不同宽带、不同机房、不同国家/地区，最优 IP 并不相同

传统做法往往是：

1. 手工收集 IP 列表
2. 用命令行工具逐个测速
3. 自己整理结果
4. 再人工修改 DNS 或配置

这个流程的问题在于，它对新手不友好，也很难长期稳定复用。

CFST-GUI 的意义，就是把这套流程产品化，让非开发者也能通过界面完成原本复杂的筛选工作。

## 4. 核心痛点解析

### 4.1 痛点一：测速工具很多，但多数偏命令行

很多测速工具本身能力不弱，但使用门槛高。普通用户常见困难包括：

- 不清楚命令参数是什么意思
- 不知道输入文件应该怎么准备
- 不知道测速结果该怎么看
- 不知道筛选出的结果怎样落地使用

CFST-GUI 把输入、配置、执行、结果查看、导出这些动作集中到了统一界面里，降低了上手门槛。

### 4.2 痛点二：单一指标无法说明真实可用性

只看 ping 或单次延迟，往往会误判线路质量。一个 IP 可能延迟不高，但：

- 丢包明显
- 路由路径不理想
- 下载速度不足
- 高峰期波动很大

CFST-GUI 把测试拆成多个阶段，分别观察延迟、追踪和文件测速结果，帮助用户避免只看单一数字。

### 4.3 痛点三：结果“测出来了”，但后续流程仍然麻烦

很多用户卡在“测速之后怎么办”：

- 如何保存结果
- 如何重复使用配置
- 如何把筛选结果更新到 DNS
- 如何在另一台机器上继续用同一套配置

CFST-GUI 内置了结果导出、配置档案、配置归档、WebDAV 备份/恢复、Cloudflare DNS 读取和自动推送等能力，让“测速”不再是孤立动作。

### 4.4 痛点四：不同使用环境需要不同交付方式

并不是每个人都在本机桌面上使用：

- 有人想在 Windows 上点开就用
- 有人想在 macOS 上本地运行
- 有人希望把工具放在 Linux 服务器或 NAS 上长期运行

CFST-GUI 针对这些场景分别提供了桌面发行版和 Docker 化的 WebUI 部署包，减少环境适配成本。

## 5. 产品定位

CFST-GUI 不是一个“万能网络加速器”，也不是一个“自动保证效果”的黑盒工具。

它更准确的定位是：

> 面向 Cloudflare/CDN 使用场景的可视化测速、筛选、导出和 DNS 落地工作台。

它主要帮助用户做三件事：

1. 更轻松地组织测速任务
2. 更直观地判断哪些候选 IP 更合适
3. 更顺畅地把结果保存、复用和推送

需要注意的是，最终效果仍然取决于你的网络环境、地理位置、上游线路质量、目标服务策略以及测试时间段。CFST-GUI 能帮你“更科学地筛选”，但不能替代真实网络环境本身。

## 6. 核心能力总览

### 6.1 可视化测速任务面板

产品提供任务看板和结果页，用户可以在界面里完成任务启动、进度查看和结果分析。

当前测速流程支持固定 4 个阶段：

- IP 池整理
- TCP 测延迟
- 追踪探测
- 文件测速

其中：

- 极速模式：执行前 3 个阶段，适合快速筛选
- 完整模式：执行全部 4 个阶段，适合更细致地看下载表现

### 6.2 多来源输入管理

候选来源支持三种常见方式：

- 远程 URL 列表
- 本地文件
- 手动输入

系统会自动清洗输入内容，跳过空行和注释，并从复杂行中提取 IP、CIDR 或域名。域名会先解析成 IP 后再参与测速。

### 6.3 更贴近实际体验的筛选指标

用户不必只看单一延迟值，还可以结合以下信息判断：

- TCP 平均延迟
- 丢包率
- 追踪探测结果
- 地区码信息
- 下载速度

这类多指标组合，更适合真实使用决策。

### 6.4 结果导出

测速结果可以导出为 CSV，方便：

- 存档留底
- 二次分析
- 与其他同事共享
- 导入到自己的后续流程

### 6.5 DNS 读取与自动推送

如果你本身就在使用 Cloudflare DNS，产品支持先从官方 API 读取当前 Zone、当前配置记录或指定子域名记录，再按需要把测速结果通过工作流、定时任务或测速后自动推送同步到 DNS 记录中。

它能够识别：

- IPv4 写入 A 记录
- IPv6 写入 AAAA 记录

DNS 页面本身只读取记录，不会修改线上 DNS；真正的推送由 Cloudflare 配置、分流规则、工作流节点或测速后自动推送列表控制。这让“测速结果”能够更快进入实际使用阶段，同时减少误点页面按钮造成的风险。

### 6.6 配置、档案与同步

CFST-GUI 不是一次性工具。它支持：

- 保存测速配置
- 保存输入源档案
- 导出/导入配置归档 ZIP
- WebDAV 备份和恢复

这对于长期维护多套线路、多套站点配置的用户尤其有帮助。

## 10. 使用发行版安装

本节面向希望直接使用成品的用户，不涉及源码编译。

### 10.1 获取发行版

从项目 Release 页面下载对应平台的安装包或压缩包。常见选择如下：

- Windows：`cfst-gui-windows-amd64.exe`
- macOS Intel：`cfst-gui-darwin-amd64.app.zip`
- macOS Apple Silicon：`cfst-gui-darwin-arm64.app.zip`
- Linux Docker 部署包：`cfst-gui-linux-amd64.tar.gz`
- Android：`cfst-gui-android-arm64-v8a-release.apk` 或 `cfst-gui-android-release.apk`

如果你拿不准该选哪一个，可以先判断自己的操作系统和 CPU 架构：

- Windows 普通 PC：通常直接选 `windows-amd64`
- Intel 芯片 Mac：选 `darwin-amd64`
- M1/M2/M3/M4 Mac：选 `darwin-arm64`
- Linux 服务器/NAS：按 CPU 架构选 `linux-amd64.tar.gz` 或 `linux-arm64.tar.gz`
- Android 手机：现代设备通常选 `arm64-v8a`，不确定时选 universal APK

### 10.2 Windows 安装步骤

1. 下载 `cfst-gui-windows-amd64.exe`
2. 将文件放到你希望长期保存的位置
3. 双击运行
4. 如果系统弹出安全提示，请确认来源后放行
5. 如果安装器提示缺少 Microsoft Edge WebView2 Runtime，请按提示打开微软下载页并安装 Runtime
6. Runtime 安装完成后，重新运行 `cfst-gui-windows-amd64.exe`
7. 进入主界面后即可开始配置输入源和测速任务

Windows 版本适合以下人群：

- 希望本机直接操作图形界面
- 不想接触 Docker
- 主要在桌面设备上手工执行测速任务

### 10.3 macOS 安装步骤

1. 下载与你设备架构匹配的 `.app.zip`
2. 解压缩得到 `CFST-GUI.app`
3. 将应用拖入“应用程序”目录，或放到你常用的位置
4. 首次打开时，如果 macOS 提示来源限制，可使用“右键打开”的方式放行
5. 成功启动后进入主界面开始使用

macOS 版本适合：

- 使用 Mac 进行日常线路测试
- 希望保留原生桌面应用体验

### 10.4 发行版首次使用建议

第一次打开后，建议按下面顺序操作：

1. 先到“输入源”页面添加候选 IP 来源
2. 在“系统配置”中确认测速参数
3. 先跑一次“极速模式”熟悉结果结构
4. 再按需要切换到“完整模式”
5. 在“当前结果”中查看结果并导出 CSV
6. 如果你确实需要落地到 Cloudflare，先用 DNS 读取页核对记录，再配置 Cloudflare 自动推送

## 11. 使用 Docker 部署 Linux WebUI

这一部分适合以下用户：

- 想在 Linux 服务器、NAS、小主机上长期运行
- 希望通过浏览器访问界面
- 希望部署方式更稳定、更容易迁移和备份

### 11.1 Docker 方式的特点

Docker 部署并不是“更高级才用”，而是更适合常驻服务场景。它的优势包括：

- 不依赖 Linux 桌面环境
- 容易迁移到另一台机器
- 数据持久化更清晰
- 更适合 7x24 小时运行

### 11.2 官方 Linux 发行包包含什么

下载 `cfst-gui-linux-amd64.tar.gz` 后，解压得到的目录中通常会包含这些文件：

- `cfst-webui`
- `Dockerfile`
- `docker-compose.yml`
- `.env.example`
- `README.md`
- `ca-certificates.crt`

这说明官方已经把 Linux WebUI 的 Docker 上下文准备好了，用户无需自己手写镜像文件。

### 11.3 标准部署步骤

### 第一步：下载并解压发行包

```bash
tar -xzf cfst-gui-linux-amd64.tar.gz
cd cfst-webui-linux-amd64
```

### 第二步：复制环境变量模板

```bash
cp .env.example .env
```

### 第三步：修改 `.env`

至少需要检查这几个变量：

```env
CFST_WEBUI_PORT=34115
CFST_WEBUI_TOKEN=change-me
CFST_VERSION=1.8.5
```

建议你至少做两件事：

- 把 `CFST_WEBUI_TOKEN` 改成自己的强口令
- 如果 `34115` 端口已被占用，就修改 `CFST_WEBUI_PORT`

说明：

- `CFST_WEBUI_TOKEN` 是 WebUI 的访问令牌
- `CFST_WEBUI_PORT` 是宿主机开放的访问端口
- `CFST_VERSION` 是镜像标签占位；如果你使用官方发行包，通常保持与当前发行包版本一致即可

### 11.4 启动服务

在解压目录中执行：

```bash
docker compose up -d --build
```

这个命令会基于发行包自带的 `Dockerfile` 构建镜像，并在后台启动服务。

### 11.5 验证服务是否启动成功

可以使用以下命令查看容器状态：

```bash
docker compose ps
```

如果容器已正常运行，就可以在浏览器中访问：

```text
http://你的服务器IP:34115
```

如果你修改过端口，请把 `34115` 替换为实际端口。

打开页面后，系统会要求输入你在 `.env` 中设置的 `CFST_WEBUI_TOKEN`。

### 11.6 数据持久化说明

官方 `docker-compose.yml` 默认会创建一个名为 `cfst-webui-data` 的 Docker volume，用于保存数据。

这意味着：

- 容器重启后，数据不会因为容器生命周期结束而丢失
- 升级版本时，通常只需替换程序与镜像，不必重建业务配置
- 做备份时，记得备份对应的 Docker volume 数据

### 11.7 停止与重启

停止服务：

```bash
docker compose down
```

重新启动：

```bash
docker compose up -d
```

查看日志：

```bash
docker compose logs -f
```

### 11.8 升级 Docker 部署

推荐升级方式如下：

1. 备份当前数据卷
2. 下载新的 Linux 发行包
3. 解压到新目录
4. 把原来的 `.env` 配置迁移过去
5. 在新目录执行 `docker compose up -d --build`
6. 验证页面可正常访问、配置仍然存在

这样做的好处是升级路径更清晰，也更方便回退。

## 12. 进阶：直接使用容器镜像

如果你已经熟悉 Docker，也可以直接使用容器镜像，而不依赖发行包里的本地构建流程。

项目工作流中约定的镜像地址格式为：

```text
ghcr.io/axuitomo/cfst-gui:<version>
ghcr.io/axuitomo/cfst-gui:v<version>
```

例如：

```bash
docker run -d \
  --name cfst-webui \
  -p 34115:34115 \
  -e CFST_WEBUI_ADDR=0.0.0.0:34115 \
  -e CFST_WEBUI_TOKEN=your-token \
  -e CFST_GUI_PORTABLE_ROOT=/data \
  -e CFST_WEBUI_ALLOWED_ROOTS=/data \
  -v cfst-webui-data:/data \
  ghcr.io/axuitomo/cfst-gui:1.8.5
```

请注意：

- 上面使用的 `1.8.5` 只是当前文档对应的示例版本号
- 真正部署时，请以你下载或计划使用的 Release 版本为准

如果你是第一次部署，更推荐优先使用官方提供的 Linux 发行包，因为它自带 `docker-compose.yml`，更省心。

## 13. 首次使用流程建议

1. 准备一份候选 IP、CIDR 或域名列表

4. 查看当前结果中的延迟、丢包、追踪信息

7. 有需要时再启用 Cloudflare 自动推送或工作流 DNS 推送节点

这样做的好处是，先用低成本方式筛掉明显不合适的候选，再对小范围结果做更细测试。

## 15. 注意事项与风险提示

### 15.1 不同网络环境下结果会不同

测速结果高度依赖：

- 地区
- 运营商
- 时间段
- 当前网络负载
- 上游链路状态

所以：

- 别人的“优选结果”不一定适合你
- 昨天测得快，不代表今天仍然快
- 同一台服务器在不同时间跑出的结果也可能不同

### 15.2 DNS 自动推送前要确认目标记录

DNS 读取页不会修改线上记录。如果你打算通过工作流、定时任务或测速后自动推送同步到 Cloudflare DNS，请务必确认：

- API Token 权限正确
- Zone ID 正确
- 记录名正确
- 分流规则和 Top N 符合预期
- 你知道本次变更会影响哪条线上记录

否则可能把生产流量切到不合适的目标上。

### 15.3 配置与归档中可能包含敏感信息

配置归档、WebDAV 备份、导出设置中，可能包含这些信息：

- Cloudflare Token
- WebDAV 凭据
- 本地路径
- 输入源路径

因此请把备份文件保存到可信环境，不要随意公开分享。

### 15.4 Docker 部署必须修改默认令牌

如果你使用 Docker WebUI，请不要直接带着默认 `change-me` 暴露到公网。

至少应做到：

- 修改令牌
- 按需修改对外端口
- 只在可信网络中暴露服务，或通过反向代理、访问控制进一步保护

## 16. 常见问题

### 16.1 我应该选桌面版还是 Docker 版？

如果你只是想在自己电脑上手工使用，优先选桌面版。

如果你希望：

- 长期运行
- 通过浏览器访问
- 部署在 NAS/服务器
- 更方便迁移和备份

那就优先选 Docker 版。

### 16.2 Linux 为什么不是原生桌面安装包？

这个项目当前对 Linux 的官方发行方式更偏向 WebUI + Docker Compose，这样更适合服务器化运行，也更利于环境统一。

### 16.3 第一次测速应该选哪个模式？

建议先从“极速模式”开始。它更快，适合初筛。

当你已经缩小候选范围，再使用“完整模式”查看下载速度，会更高效。

### 16.4 这个工具会自动帮我保证最佳线路吗？

不会。它的作用是帮助你更系统地测试和筛选候选目标，而不是替你承诺最终效果。

## 17. 总结

CFST-GUI 的价值，不在于“把一个命令换成按钮”，而在于把一整套零散的测速、筛选、导出和 DNS 落地流程，整理成了一个普通用户也能上手的工作台。

如果你的目标是：

- 更容易地测试 Cloudflare/CDN 候选 IP
- 更直观地筛选结果
- 更稳定地沉淀配置与结果
- 更方便地在桌面或 Linux 服务器上长期使用

那么 CFST-GUI 是一个很适合入门和持续使用的选择。

对于最终用户，推荐路线很简单：

- Windows/macOS 用户：直接下载发行版使用
- Linux/NAS/服务器用户：直接使用官方 Docker Compose 发行包部署

这样既能降低使用门槛，也能保留足够完整的功能能力。
